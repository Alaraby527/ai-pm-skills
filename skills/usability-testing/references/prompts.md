# 可用性测试提示词合集

## 1. 测试设计阶段

### 1.1 测试方案设计提示词

```
我要做一次可用性测试，产品是【产品/功能名称】。

测试目标：
1. 【目标 1】
2. 【目标 2】
3. 【目标 3】

产品背景：
【简要描述产品是什么、当前处于什么阶段】

目标用户：
【描述目标用户画像】

请帮我设计完整的测试方案：
1. 推荐的测试方法（形成性/总结性、远程/线下、任务测试/探索性）
2. 建议测试人数及理由
3. 用户招募标准（纳入/排除）
4. 测试时长和流程安排
5. 需要准备的物料和工具
6. 注意事项和常见坑
```

### 1.2 测试任务设计提示词

```
我要为【产品/功能】设计可用性测试任务。

测试目标：【描述测试目标】
目标用户：【描述用户画像】
测试时长：45-60 分钟

请帮我设计 5-8 个测试任务，要求：
1. 基于真实使用场景，不是"测试功能"而是"完成任务"
2. 任务难度递进（从简单热身到复杂任务）
3. 每个任务包含：场景描述、用户目标、成功标准、预计时长
4. 任务描述避免引导性（不说"点XX按钮"，说"找到XX信息"）
5. 覆盖测试目标的所有关键方面
6. 每个任务后面加一句【设计说明】解释为什么这么设计

产品核心功能/页面：
【列出主要功能和页面】
```

### 1.3 任务难度评估与优化提示词

```
这是我设计的可用性测试任务清单：
【粘贴任务清单】

请帮我审查和优化：
1. 任务描述有没有引导性？（有没有暗示用户怎么操作）
2. 任务是不是基于场景而不是基于功能？
3. 任务难度顺序合理吗？（应该从易到难）
4. 总时长会不会太长？（建议 30-40 分钟任务时间）
5. 任务覆盖度够吗？有没有遗漏重要的功能/场景？
6. 成功标准清晰吗？（能判断用户有没有完成）
7. 哪些任务可以合并？哪些可以删掉？

输出：修改建议 + 优化后的任务清单
```

## 2. 测试执行阶段

### 2.1 测试开场脚本生成提示词

```
请帮我写一份可用性测试的开场脚本，用于测试员照着念。

产品类型：【产品类型】
测试形式：线下 / 远程
测试时长：约 60 分钟

脚本需要包含：
1. 自我介绍
2. 测试目的说明
3. 出声思维法的说明（简单明了，让用户知道该怎么做）
4. "不是测试您，是测试产品"的心理建设
5. 录音/录屏授权说明
6. 保密承诺
7. 激励说明（可选）
8. 开场后进入背景了解的过渡

要求：
- 语气自然、亲切，不生硬
- 不太长（控制在 2 分钟内能说完的量）
- 口语化，不像在读稿子
```

### 2.2 追问话术参考提示词

```
我是可用性测试的测试员，请问在以下场景中我应该怎么说？

场景 1：用户操作过程中突然不说话了
场景 2：用户问我"这样对吗？"
场景 3：用户卡住了，好像进行不下去了
场景 4：用户说"这个设计得好烂啊"（情绪性评价）
场景 5：任务完成了，用户看着我等我说话
场景 6：用户问了一个功能相关的问题，我该不该回答？

每个场景给 2-3 种问法，并说明背后的原则。

原则回顾：
- 不指导、不纠错、不解释
- 用问题回答问题
- 鼓励用户多说自己的想法
```

## 3. 分析输出阶段

### 3.1 问题整理与分级提示词

```
以下是 X 场可用性测试中发现的问题汇总：
【列出所有问题，每条标注：在哪个任务、哪个用户遇到的、具体表现】

请帮我整理：
1. 去重合并：相同/相似的问题合并
2. 严重程度分级：致命（任务失败）/ 严重（很困难）/ 一般（有障碍）/ 轻微（体验不好）
3. 频次统计：每个问题有多少用户遇到
4. 按优先级排序（先按严重程度，同级按频次）
5. 每个问题给一个简短的改进建议（方向即可，不用太细）

输出格式：表格，包含问题描述、严重程度、出现频次、涉及用户、改进建议。
```

### 3.2 启发式评估提示词

```
以下是可用性测试中发现的 X 个问题：
【列出问题清单】

请用 Nielsen 十项启发式原则对这些问题进行分类：
1. 系统状态可见性
2. 系统与现实世界匹配
3. 用户控制与自由度
4. 一致性与标准化
5. 错误预防
6. 识别优于回忆
7. 使用灵活性与效率
8. 美感与简约设计
9. 帮助用户识别、诊断和恢复错误
10. 帮助与文档

要求：
1. 每个问题归类到对应的启发式原则（可以多选）
2. 简单解释为什么归到这一类
3. 统计每类原则有多少问题（哪类问题最多）
4. 最后给出整体评价：这款产品最大的可用性问题出在哪个层面？
```

### 3.3 测试报告生成提示词

```
请基于以下测试数据，生成一份可用性测试报告：

【测试背景：】
【测试用户：X 人，画像分布...】
【核心发现（原始素材）：】
【问题清单：】
【任务完成数据：】

报告结构：
1. 测试概况（背景、目标、方法、用户）
2. 核心发现（3-5 条，每条有数据+分析+建议）
3. 可用性指标（完成率、完成时间、满意度）
4. 问题清单（按严重程度分级排序）
5. 改进建议与优先级
6. 正面发现

要求：
- 问题导向，结论明确
- 有数据支撑，不主观
- 建议具体可执行
- 语言专业但不晦涩
```

## 4. 快速评估（不用用户测试的方法）

### 4.1 启发式评估（专家走查）提示词

```
请用 Nielsen 十项启发式原则，对【产品/功能/页面】做一次启发式评估。

【产品/页面描述：】
【截图/操作流程描述：】

请逐一检查十项原则，输出：
1. 每项原则的检查结果（是否有问题）
2. 发现的具体问题（描述位置和表现）
3. 问题严重程度（严重/一般/轻微）
4. 改进建议

最后给出一个总体评价和改进优先级排序。
```

### 4.2 SUS 问卷设计提示词

```
我想在可用性测试结束后让用户填 SUS（系统可用性量表）问卷。

请提供：
1. 标准的 SUS 问卷（10 道题，标准版本）
2. 计分方法和解释
3. SUS 分数的参考标准（多少分算好）
4. 使用注意事项
5. 有没有中文版的翻译版本
```
